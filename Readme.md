# SAWM - запуск лабораторных работ (Lab1-Lab7)

## 1) Что нужно установить
- PHP 8.1+ (в проекте проверено на PHP 8.4).
- MySQL 8+.
- Доступ к CLI-командам `php` и `mysql`.

## 2) Общие тестовые данные
- Администратор: `admin / admin123`
- Менеджер: `manager / manager123` (используется начиная с Lab5)

## 3) Общий порядок запуска любой лабораторной
1. Перейдите в папку нужной лабораторной:
   - `cd LabN`
2. Инициализируйте БД:
   - `mysql -u root -p < init.sql`
3. Запустите встроенный PHP-сервер:
   - `php -S localhost:8000`
4. Откройте в браузере:
   - `http://localhost:8000/index.php`

## 4) Пошаговый сценарий по каждой лабораторной

Лабораторная работа 2. Методы предотвращения SQL-injection в веб приложениях
Предварительные условия
1. Создайте веб-страницу – доступ к которой ограниченный.
2. Эта страница будет закрытым пространством для администратора веб-контента, доступ к
которому осуществляется через проверку подлинности (аутентификации).
3. Создайте еще одну страницу, содержащую форму аутентификации, которая имеет 2
элемента управления - для логина и пароля.
4. Создайте базу данных, содержащую таблицу «user», в которой будут храниться данные со
следующей структурой: id, login, password.
Требования лабораторной работе
1. Создайте 7 новых записей в таблице «user».
2. Попробуйте использовать SQL-инъекцию, чтобы войти в панель администрирования
(закрытое пространство администратора), не использовав правильные данные существующих
аккаунтов (посмотрите дополнительно
https://www.owasp.org/index.php/Testing_for_SQL_Injection_(OTG-INPVAL-005))
3. Создайте client-side и server-side скрипты, которые будут бороться с возможностью ввода
кода SQL, в элементы управления формы, путем проверки данных.
4. Проверьте безопасность созданного мини-приложения

Лабораторная работа 3. Предотвращение атаков инъекций XSS-кода
Предусловия
1. Создайте базу данных „guest”, содержащую одну таблицу - „guest”, которая будет
содержать данные со следующей структурой: id, user, text_message, e_mail,
data_time_message;
2. Создайте мини-приложение типа guest-book, которая позволит хранение введенных
пользователем данных в форму и которая потом выводит на экран все записи из таблицы
„guest”.
Требования лабораторной работе
1. Продемонстрируйте как можно использовать язык JavaScript для XSS атак, через контролы
созданной формы, непроверяя входные данные, перенапрвив пользователя на другой сайт
или вставив вместо всей страницы (все что было между <body>…</body>) спорную
фотографию. То есть необходимо вместо ожидаемого контента вывести другой,
неожиданный для пользователя.
2. Добавьте скрипты проверок чтобы небыло возможностей XSS атак.

Лабораторная работа 4. Защита конфиденциальных данных
Требования к лабораторной работе
1. Используйте созданную ранее приложение в 1-й или 3-й лабораторной работе;
2. Выведите на экран содержимое таблицы „user”;
3. Проанализируйте уязвимость – когда пароли хранятся в открытом виде, без применения
хэширования;
4. Объясните преподавателю, как рекомендуется формировать и хранить пароли. Приведите
примеры уязвимых и надежных паролей;
5. Объясните преподавателю, почему хэш-функции MD5, SHA1 уже считаются уязвимыми и
почему. Какие атаки можно выполнить на пароли, если они «захэшированны» с помощью
MD5;
6. Примените рекомендуемые сегодня надежные хэш-функции с «солью» к паролям;
7. Объясните преподавателю, почему вы применили данную функцию, а также
продемонстрируйте, как пароли хранятся в базе данных. Объясните, в чем преимущество
хранения паролей «скрытым» способом. Объясните, какова роль «соли».

Лабораторная работа 5. Контроль доступа к веб приложению
Требования к лабораторной работе
1. Используйте мини-приложение созданную в 1-ой лабораторной и продемонстрируйте что
если пользователь знает URL ресурса, содержащий пано администрирования, может сразу
зайти туда, без аутентификации.
2. Внедрите правильно принципы использования сеcсии в веб приложение и докажите что
больше невозможно зайти в пано администрирования без успешной аутентификации –
аутентификация обязательна;
3. Дополнительно создайте аккаунт для пользователя «менеджер» (добавьте а таблицу «user»
дополнительное поле «роль» - у вас сейчас в поле роль должны быть 2 возможных значения
– «администратор» и «менеджер»). Пользователь с ролью «менеджер» должен иметь доступ
к другому пространству для работы и другим функционалом (пусть будут 2 функции доступны
этому виду пользователя), к которому он имеет доступ, также, после успешной
аутентификации.
4. Дополните БД необходимыми таблицами так чтобы была возможность авторизации
пользователей (используя переменные сесcии), так чтобы после аутентификации
пользователя, в зависимости от его роли он был перенаправлен на то пано (страницу) которая
должна быть ему доступна, соответственно его роли.
5. Проверьте, имеют ли пользователи доступ к разрешенным функциям и правильно ли
реализован контроль доступа на основе ролей и разрешений. Внедрите функции для
«менеджера» по просмотру данных учетных записей, изменению данных или их удалению,
но эти функции не должны быть доступны простому пользователю.
6. С обеих закрытых пространств должна существовать возможность выхода со «сбросом» на
страницу аутентификации.
7. Продемонстрируйте преподавателю, как работать с приложением для пользователей с
разными ролями, пройдя весь цикл событий для пользователей обеих ролей.

Лабораторная работа 6. Журналирование действий пользователей в целях безопасности
(logging)
Требования к лабораторной работе
1. Используйте ранее созданное приложение в 4-5-й лабораторных работах.
2. Создайте файл логов для хранения в нем действий пользователей: кто, когда вошел, какие
действия выполнил по отношении с приложением, ... когда вышел. (посмотрите пример с
лекции для добавлени и других деталей).
3. Добавьте пользователю «администратор» опцию, которая доступна ему только после
авторизации и проверки роли, через которую он может посмотреть содержимое лог-файла.
4. Объясните преподавателю необходимость регистрации действий пользователей,
необходимость хранения обезличенных, неконфиденциальных данных в файле журнала,
необходимость анализа содержимого файлов журнала, а также ротацию/изменение
журналов.

Лабораторная работа 7. Манипулирование ошибками и их протоколирование
Требования к лабораторной работе
1. Обработайте ошибки и правильно информируйте пользователя о том что он может дальше
сделать в приложении;
2. Все ошибки и предупреждения запротоколируйте – создавая текстовой файл;
3. Объясните преподавателю, почему текст ошибки приложения, видимый простому
пользователю, может быть уязвимостью для веб-приложения.
4. Создайте файл, который будет отвечать за генерацию страницы-«убежища» для
пользователя в случае ошибки. Объясните преподавателю необходимость этой страницы,
почему она должна быть статической (а не динамически генерируемой) и почему в её
содержимом должна быть хотя бы одна ссылка, чтобы пользователь мог вернуться в вебприложение.
5. Добавьте опцию администратору при помощи которой администратор сможет посмотреть
файл ошибок (только после проверки роли).
