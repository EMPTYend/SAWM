# SAWM: Полный README (Lab1-Lab7)

## О проекте
`SAWM` — это учебное PHP/MySQL веб-приложение, которое последовательно развивается от уязвимой версии к защищенной.

Логика построена по шагам:
1. Сначала создается уязвимая база (Lab1).
2. Затем поэтапно закрываются уязвимости (SQLi, XSS, хранение паролей).
3. После этого добавляются контроль доступа, логирование действий и безопасная обработка ошибок.

Итог к Lab7: приложение с аутентификацией, RBAC, аудитом действий и централизованным логированием ошибок.

---

## Что нужно для запуска
- Windows + XAMPP (рекомендуется).
- PHP 8.x с включенным `pdo_mysql`.
- MySQL/MariaDB (через XAMPP).

Рекомендуемые бинарники:
- `C:\xampp\php\php.exe`
- `C:\xampp\mysql\bin\mysql.exe`

Тестовые аккаунты:
- `admin / admin123`
- `manager / manager123` (начиная с Lab5)

## Быстрый запуск любой лабораторной
1. Запусти `C:\xampp\xampp-control.exe`.
2. Нажми `Start` у `MySQL`.
3. Открой `cmd` и выполни:

```bat
cd "C:\Users\vipia\OneDrive\Рабочий стол\SAWM\LabN"
"C:\xampp\mysql\bin\mysql.exe" -u root < init.sql
"C:\xampp\php\php.exe" -S localhost:800N
```

Где `N`:
- `1` для Lab1
- `2` для Lab2
- `3` для Lab3
- `4` для Lab4
- `5` для Lab5
- `6` для Lab6
- `7` для Lab7

Открывать в браузере:
- `http://localhost:800N/index.php`

## Что выполнено от начала до конца

## Lab1: Базовая уязвимая версия
Что добавлено:
- Базовая форма логина (`index.php`, `login.php`).
- Таблица `user` в БД (`init.sql`, 7 пользователей).
- Страница `admin.php`.

Для чего:
- Создать исходную точку с уязвимостями для последующего усиления.

Как работает:
- Логин проверяется SQL-запросом, собранным конкатенацией строк.
- `admin.php` доступна по прямому URL без сессии.
- Пароли хранятся в открытом виде.

Как проверить:
1. Открыть `http://localhost:8001/admin.php` без логина.
2. В форме входа попробовать SQL payload `admin' -- `.
3. Убедиться, что вход возможен без корректного пароля.

## Lab2: Защита от SQL Injection
Что добавлено:
- Клиентская валидация (`validation.js`).
- Серверная валидация в `login.php` (regex для login/password).
- Prepared statements (`PDO::prepare`) вместо динамического SQL.

Для чего:
- Исключить выполнение SQL-кода, введенного пользователем.

Как работает:
- Недопустимые символы блокируются на фронте и бэке.
- Запрос выполняется только с параметрами `:login`, `:password`.

Как проверить:
1. Ввести `admin' -- ` в поле `Login`.
2. Ожидание: ввод блокируется или возвращается ошибка `Input contains blocked symbols.`.
3. Вход с валидными `admin / admin123` должен работать.

---

## Lab3: Защита от XSS в guest-book
Что добавлено:
- Таблица `guest`.
- Страница `guestbook.php` с формой и выводом сообщений.
- Клиентская проверка (`guest_validation.js`).
- Серверная фильтрация/валидация:
  - проверка имени и email,
  - `strip_tags()` перед записью,
  - `htmlspecialchars()` при выводе.

Для чего:
- Предотвратить выполнение внедренного JavaScript в сообщениях.

Как работает:
- HTML/JS payload блокируется или обезвреживается.
- Даже сохраненные данные отображаются только в безопасном виде.

Как проверить:
1. В `Message` ввести `<script>alert(1)</script>`.
2. Ввести `<img src=x onerror="alert(1)">`.
3. Ожидание: скрипт не выполняется, алерт не появляется.

## Lab4: Защита конфиденциальных данных (пароли)
Что добавлено:
- Хранение паролей в виде хешей Argon2id (`init.sql`).
- Проверка пароля через `password_verify()` в `login.php`.
- Просмотр таблицы пользователей в `admin.php`.

Для чего:
- Убрать хранение паролей в plain text.

Как работает:
- В БД лежат только хеши (`$argon2id$...`).
- Сравнение идет не по строке пароля, а по проверке хеша.

Как проверить:
1. Войти `admin / admin123`.
2. Открыть `admin.php` и проверить, что в колонке пароля хеши.
3. Убедиться, что `wrong`-пароль не проходит.

---

## Lab5: Аутентификация, сессии, роли (RBAC)
Что добавлено:
- Безопасные сессии (`auth.php`):
  - `HttpOnly`, `SameSite`,
  - регенерация session ID,
  - корректный logout.
- Роли `administrator` и `manager` в таблице `user`.
- Таблицы RBAC: `role`, `permission`, `role_permission`.
- Ролевой редирект после логина (`admin.php` / `manager.php`).
- Проверки `require_role()` и `require_permission()`.
- Функции менеджера: просмотр/изменение/удаление аккаунтов.

Для чего:
- Запретить доступ к закрытым страницам без входа.
- Разделить доступ по ролям и разрешениям.

Как работает:
- Без сессии пользователь отправляется на `index.php`.
- Для каждой закрытой страницы проверяется роль/permission.

Как проверить:
1. Открыть `admin.php` и `manager.php` без входа.
2. Войти как `admin` и проверить доступ только к админским страницам.
3. Войти как `manager` и проверить доступ к `accounts.php`.
4. Проверить logout: после выхода закрытые URL снова недоступны.

## Lab6: Журналирование действий пользователей
Что добавлено:
- `logger.php` и файл `logs/actions.log`.
- Логирование событий:
  - успешный/неуспешный вход,
  - отклонение валидацией,
  - отказ доступа,
  - открытие панелей,
  - просмотр/редактирование/удаление аккаунтов,
  - logout.
- Страница `view_logs.php` только для администратора (`logs.view`).

Для чего:
- Аудит и расследование инцидентов.
- Контроль действий пользователей по времени.

Как работает:
- Каждое событие пишется JSON-строкой с метаданными пользователя, IP и временем.
- Просмотр логов ограничен ролью/разрешением.

Как проверить:
1. Сделать успешный и неуспешный логин.
2. Открыть `accounts.php`, выполнить edit/delete (под manager).
3. Под admin открыть `view_logs.php` и увидеть соответствующие события.
4. Под manager попробовать открыть `view_logs.php` и получить отказ.

## Lab7: Безопасная обработка ошибок и error logging
Что добавлено:
- Глобальный обработчик ошибок (`error_handler.php`):
  - `set_error_handler()`,
  - `set_exception_handler()`,
  - `register_shutdown_function()`.
- Отключен показ внутренних ошибок пользователю (`display_errors=0`).
- Статическая safe-page: `safe.html`.
- Лог ошибок: `logs/errors.log`.
- Страница `view_errors.php` только для администратора (`errors.view`).

Для чего:
- Не раскрывать внутренние детали приложения обычному пользователю.
- Сохранять диагностику для администратора.

Как работает:
- Внутренние ошибки логируются в `errors.log`.
- Пользователь перенаправляется на `safe.html` без stack trace.

Как проверить:
1. Запустить Lab7 и остановить MySQL в XAMPP.
2. Открыть страницу, которая идет в БД (например `guestbook.php`).
3. Ожидание: редирект на `safe.html`.
4. Запустить MySQL обратно, войти как admin и открыть `view_errors.php`.
5. Убедиться, что запись об ошибке есть в `errors.log`.

## Частые проблемы
Проблема: `'mysql' is not recognized ...`
- Используй полный путь: `"C:\xampp\mysql\bin\mysql.exe" -u root < init.sql`.

Проблема: `Temporary Safe Page` сразу после запуска
- Обычно это ошибка подключения к БД или отсутствие `pdo_mysql` у текущего PHP.
- Запускай сервер через `"C:\xampp\php\php.exe" -S localhost:800N`.
